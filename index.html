Design System

A comprehensive reference for engineers, designers, and system integrators building on top of the Deepfake Defense program. This document covers every design token, component specification, layout pattern, and usage convention in the system. Use it as the source of truth for extending the site, building the Interactive Dashboard, or integrating detection signals into your existing security tooling.

Section 01

Design Tokens variables

All design decisions are encoded as CSS custom properties on :root. This is the single source of truth. Never hardcode values. Reference these tokens in production code and in any downstream consumer (dashboard, plugin, SDK).

Reference tokens via var(--token-name) in all CSS. Keeps theming consistent and makes global changes a single-line edit.
Hardcode hex values like #4cd2ff in component CSS. Use the token. Multiplied tokens break instantly.
Engineering rule: If you find yourself typing a hex, rgb, or raw pixel value in CSS, stop. The value does not exist in the design system. Add a new token or use an existing one. This applies to all team members — frontend, dashboard, email templates, or embedded widgets.

Section 02

Typography fonts

Two font families, strictly scoped. Inter drives all UI and body copy. JetBrains Mono is reserved for data, code, technical labels, and forensic readouts. Never use a system font fallback for body text — the whole aesthetic depends on these choices.

Inter (Display & Body)

Weights 400, 500, 600, 700, 800. The entire UI surface is Inter. Use 800 for h1, 700 for h2-h4, 600 for subheads, 500 for body, 400 for captions.

The quick brown fox (800) The quick brown fox (700) The quick brown fox (600) The quick brown fox (500) The quick brown fox (400)
Token: --font-sans for UI, --font-display for headings. Always use letter-spacing: -0.015em on headings and -0.025em on h1.

JetBrains Mono (Code & Data)

Weights 400, 500, 700. Reserved for forensic readouts, signal tables, detection scores, timeline dates, and technical labels. Never use for prose.

lip_sync_delta: 42ms (700) confidence: 0.87 · FACE-014 (500) Target: >90% recognition rate (400)
Token: --font-mono. Also applies to the code and kbd elements globally (styled in base CSS).

Section 04

Color palette + roles

Three accent colors — cyan, violet, magenta — on a midnight indigo field. These are not swatches to choose from; they are roles with specific semantic meaning. Use them deliberately.

Golden rule of the palette: cyan is primary action, violet is NIST/standards/authority, magenta is alert/synthetic/warning. Never use cyan for a warning badge or magenta for a primary button. The role is locked.

Use IntersectionObserver with threshold: 0.08 and rootMargin: '0px 0px -40px 0px'. Animate once, unobserve immediately.
Re-trigger animations on every scroll. Use CSS-only scroll-driven animations (unsupported across browsers).

Adding a new page

  2. Update .nav-links a.active on the current page
  3. Add header.hero with eyebrow + h1 + lede
  4. Add section.section blocks with .section-header

  1. Add a <button class="nist-chip"> element
  2. Set the five data-nist-* attributes
  3. Wrap in <span class="nist-chip-row"> if inline

Adding a quiz question

  1. Locate the data-quiz JSON on training.html
  2. Add a new object to the array with q, options[], answer, explain
  3. No HTML changes needed. The JS parses the JSON automatically

CSS Convention Summary

ConventionRule
NamingBEM-ish: block__element--modifier. Components use flat classes (.card, .nist-chip).
VariablesAlways reference var(--token). Never hardcode values.
AnimationsCSS-only except for IntersectionObserver. No JS-driven animation libraries.
States:hover → translateY lift + border highlight. :active → reset. :focus-visible → cyan outline.
TouchAll interactive elements have cursor: pointer. No separate touch styles — the existing hover states degrade gracefully.

Common Mistakes to Avoid

Always use --font-mono for forensic/data readouts and --font-sans for everything else.
Apply JetBrains Mono to body paragraphs or general UI copy. It breaks readability.
Use the eyebrow component at the top of every major section for hierarchy.
Create custom badge/label components. The .eyebrow is the only section label component.
Test accent color roles: cyan for actions, violet for standards, magenta for warnings.
Swap accent roles. A magenta CTA or a cyan warning badge breaks the semantic system.
Add new pages by following the 7-layer page architecture template.
Add custom layout containers, custom breakpoints, or ad-hoc spacing. Use .container and .section.

Git Diff Conventions consistent change patterns

Every design system change follows a predictable diff pattern. Use these templates to write clear PRs that are easy for reviewers to validate. A diff should tell a story — not just "changed some values."

Pattern A: Rename a design token

DO — 3 files changed, same diff pattern
+--cyan-action: #38bdf8;
- --cyan: #4cd2ff;

1. style.css :root
2. designsystem.html tokens table
3. design-tokens.json accent block
DON'T — only changed in CSS, missed docs
- --cyan: #4cd2ff;
+ --cyan: #38bdf8;

"I changed the cyan — the rest will update itself"
→ No. Update all 3 files in one commit.
Rule: A token rename touches exactly 3 files: style.css, designsystem.html (tokens table + swatch gallery), design-tokens.json. The Next.js globals.css should be updated in a separate follow-up or the same commit with a note. A PR that changes a token in only 1 of these files is rejected.

Pattern B: Add a new component

+ css/style.css — component block with comment header
+ designsystem.html — .component-preview with live example + spec table + states
+ designsystem.html — state matrix entry in Section 05b
+ designsystem.html — usage example in Section 06b (if a composition uses it)
+ target HTML page — first production use of the component

// PR description: "Adds ThreatCard component. 4 files changed. Demo on threat-landscape.html page-2."
Rule: Minimum 4 files for a new component. CSS + design system spec + state matrix + one production page. A component without a design system spec does not exist.

Pattern C: Change a breakpoint value

- @media (max-width: 900px) { ... }
+ @media (max-width: 860px) { ... }

+ design-tokens.json — update breakpoints.tablet.value
+ designsystem.html — update responsive breakpoint table

// PR description: "Shrink tablet breakpoint: 900→860px. Catches iPad Mini width gap reported in issue #42."
Rule: A breakpoint change must be validated at the new value on real devices. Attach screenshots or a BrowserStack URL to the PR. Do not change breakpoints without team review.

Pattern D: Fix component state (e.g., add focus-visible)

+ style.css — .btn:focus-visible { outline: 2px solid var(--cyan); outline-offset: 2px; }

+ designsystem.html — update Button state matrix row (focus-visible: "now handled")

// PR description: "Add focus-visible ring to .btn. Closes a11y gap #17."
PR template summary: Every design system PR should answer: (1) What changed? (2) Which files were updated? (3) Is the token JSON in sync? (4) Are the design system docs updated? (5) Was this tested at both breakpoints? If the answer to any of (3)-(5) is "no", the PR goes back to draft.

JS Contracts & Data Attributes main.js patterns

The site's interactivity is driven by data attributes on HTML elements. Every attribute is a contract: if you use it in the markup, the JS behavior is guaranteed. Do not re-implement these patterns.

[data-count] — Animated stat counters

AttributeTypeRequiredExample
data-countNumberYesdata-count="240" or data-count="2.4"
data-suffixStringNodata-suffix="M" renders as 2.4M

Animate on scroll via IntersectionObserver (threshold 0.4). Uses cubic ease-out over 1600ms. Falls back to immediate render if IntersectionObserver is unavailable. See main.js:34-60.

[data-fusion-bar] — Detection fusion bars

AttributeTypeExample
data-fusion-barNumber (0–1)data-fusion-bar="0.78" → bar fills to 78%
data-fusion-valElement to display valueText content set to 0.78
data-fusion-rowContainer grouperScopes the querySelector for the value element

Triggered on scroll via IntersectionObserver (threshold 0.3). 200ms delay, CSS transition 1.2s cubic-bezier(0.4, 0, 0.2, 1). See main.js:157-175.

[data-quiz] — Training quiz

JSON array directly in the HTML attribute. Schema per object:

PropertyTypeDescription
qStringQuestion text (JSON-escaped)
optionsString[4]Exactly 4 answer options
answerNumber (0–3)Index of the correct option
explainStringExplanation shown after answering

JS renders, locks after answer, scores, provides feedback, and shows restart. See main.js:178-253.

[data-form] — Contact form handler

Simple client-side handler. Prevents submit default, shows .form-status with thank-you message, resets form. No server endpoint (main.js:255-264). Wire to real endpoint for production.

[data-scan-id] + [data-verdict] — Face scanner

AttributeBehavior
data-scan-idTextContent overwritten with faceData[idx].id on thumb switch or auto-rotate
data-verdictInner .row b and .row span:last-child updated with confidence, lipSync, noise. .bar-fill width animated.

faceData schema: { id: string, confidence: number, lipSync: string, noise: "normal" | "anomalous" }. Thumbnails use .face-thumb class + .active state. Auto-rotate interval: 5s. Manual click pauses rotation (main.js:82-147).

ARIA roles & attributes

  • .nist-chip → role="button", tabindex="0", aria-expanded
  • .nist-popover → role="dialog", aria-hidden
  • .nav-toggle → aria-label="Toggle menu"
  • External links → target="_blank" rel="noopener noreferrer"
  • Form inputs → label via for/id or wrapping

Keyboard navigation

ComponentKeyBehavior
.nist-chipEnter / Space / EscapeEnter/Space opens popover. Escape closes. Global Esc listener closes any open popover (main.js:359-368).
.nav-toggleEnter / SpaceToggles mobile menu open/close (main.js:24-32).

Reduced motion

A global @media (prefers-reduced-motion: reduce) at style.css:1123 kills all animations and transitions with !important. Covers: scan line, pulse dots, page fade, card hovers, scroll reveals, fusion bar reveals, face crossfade, parallax. Body animation is disabled. Any new animation must be covered by this query.

Known gaps (future work)

  • Skip-to-content link — currently absent from all pages. Should be first focusable element.
  • Focus-visible — only on .nist-chip. Should extend to buttons, quiz options, nav links.
  • Landmarks — semantic <nav>, <header>, <main>, <footer> are correct. Maintain this structure on new pages.
